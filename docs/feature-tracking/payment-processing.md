# Payment Processing

## Provider

**PayOS** (Vietnamese payment gateway) — integrated via `@payos/node` v2.0.5 and `@payos/payos-checkout` v1.0.8.

SDK wrapper: `app/.server/payment/provider.ts`

Exposes:

- `createPaymentLink(data)` → `{ checkoutUrl, paymentLinkId }`
- `getPaymentLink(orderCode)` — query payment status
- `cancelPaymentLink(orderCode, reason)` — cancel link
- `verifyWebhookData(body)` — verify webhook signature

## Database

### `orders` table (`schema.ts:248`)

| Column            | Type                      | Notes                                                              |
| ----------------- | ------------------------- | ------------------------------------------------------------------ |
| `id`              | `integer` PK              | Auto-generated                                                     |
| `order_code`      | `integer` UNIQUE          | Generated by server (`crypto.randomInt(100_000_000, 999_999_999)`) |
| `payment_link_id` | `varchar(255)`            | PayOS payment link ID                                              |
| `user_id`         | `integer` FK → `users.id` | Who placed the order                                               |
| `plan`            | `varchar(50)`             | `'Lite'` or `'Pro'`                                                |
| `amount`          | `integer`                 | In cents (e.g. 99 for $0.99)                                       |
| `status`          | `varchar(20)`             | `'PENDING'` or `'PAID'`                                            |
| `expires_at`      | `timestamp`               | Computed at creation: `now() + 30 days`                            |
| `created_at`      | `timestamp`               | Auto-set                                                           |
| `updated_at`      | `timestamp`               | Auto-set                                                           |

### `users` table — `role` column

| Value       | Meaning                                                                      |
| ----------- | ---------------------------------------------------------------------------- |
| `'learner'` | Default — non-paying user                                                    |
| `'lite'`    | Purchased Lite plan — has active 30-day subscription                         |
| `'pro'`     | Purchased Pro plan — has active 30-day subscription                          |
| `'staff'`   | Manually assigned admin — has access to admin panel (not related to payment) |

Also: `subscription_ends_at` (nullable timestamp) — tracks when the current subscription period ends. `null` means no active plan (learner or staff).

### `chat_messages` table (`schema.ts:249`)

| Column    | Type                      | Notes                               |
| --------- | ------------------------- | ----------------------------------- |
| `id`      | `integer` PK              | Auto-generated                      |
| `user_id` | `integer` FK → `users.id` | Who sent the message                |
| `date`    | `varchar(10)`             | ISO date string (`YYYY-MM-DD`)      |
| `count`   | `integer`                 | Number of AI messages sent that day |

Unique constraint on `(user_id, date)`. Used by AI chat quota system: lite = 20/day, pro = unlimited.

## Complete Flow

### 1. Upgrade Entry Point

- `MainLayout.tsx` — Sidebar shows a _"Go Pro"_ card for users with `role === 'learner'` only
- Clicking _"Upgrade Now"_ sets `isPricingOpen = true`, opening `PricingModal`

### 2. Pricing Modal (`PricingModal.tsx`)

| Plan | Price           | Features (UI copy)                                                                     |
| ---- | --------------- | -------------------------------------------------------------------------------------- |
| Free | $0              | Theory lessons, free labs, limited practical labs, limited AI chat                     |
| Lite | $0.99 / 30 days | Unlimited learning paths, limited advanced labs, private environments, limited AI chat |
| Pro  | $4 / 30 days    | Unlimited advanced labs, private environments, unlimited AI tutor, priority support    |

Note: These features are **not enforced** in code — the plan selection only determines the amount charged and the role granted.

Selecting Lite or Pro opens `CheckoutModal`.

### 3. Checkout Modal → Create Payment Link

- `CheckoutModal.tsx` converts price to cents and POSTs to `/payment` with `{ intent: 'create-checkout', plan, amount }`
- Server action (`app/routes/payment.ts`):
    1. Authenticates user
    2. Generates `orderCode` (last 9 digits of timestamp + last digit of user ID)
    3. Calls `payOS.createPaymentLink()` with `orderCode`, `amount`, `description` (`'LITE SUB'` / `'PRO SUB'`), `returnUrl`, `cancelUrl`, `buyerName`, `items`
    4. Computes `expiresAt = now() + 30 days`
    5. Inserts order row with `status = 'PENDING'`, `expiresAt`
    6. Returns `{ checkoutUrl }`
- Frontend redirects via `window.location.href = data.checkoutUrl`

### 4. PayOS Checkout Page

User completes payment on PayOS's hosted page. PayOS then:

- Sends **server-to-server webhook** POST to `/payment-webhook`
- Redirects browser to `returnUrl` (`/payment/success?orderCode=...`) or `cancelUrl` (`/payment/cancel?orderCode=...`)

### 5. Webhook Processing (`PaymentWebhook.ts`)

1. Verify webhook body via `payOS.verifyWebhookData(body)` — 401 if invalid
2. Check `webhookData.code === '00'` (success)
3. Look up order by `webhookData.orderCode`
4. **Idempotency**: skip if order status is already `'PAID'`
5. **DB transaction**:
    - Update order `status = 'PAID'`
    - Look up current `subscriptionEndsAt` on user
    - Compute new expiry: `MAX(currentEnd, now()) + 30 days` (extends on renewal)
    - Update user `role = 'lite'` (Lite plan) or `role = 'pro'` (Pro plan)
    - Update user `subscriptionEndsAt = newEnd`
6. Return `{ code: '00', desc: 'success' }`

### 6. Success / Cancel Pages

- **Success** (`PaymentSuccess.tsx`): Green checkmark, order code, link to dashboard
- **Cancel** (`PaymentCancel.tsx`): Red X, order code, _"Try Again"_ link to home

## Subscription Expiry & Downgrade

Handled in `app/middleware/auth.ts` on every authenticated request:

- **Fast path** (session cache): reads `subscriptionEndsAt` from session, downgrades role to `'learner'` if expired
- **Fallback path** (DB query): reads user row, downgrades `role` to `'learner'` in both context and DB if expired
- Only subscription roles (`'lite'`, `'pro'`) are downgraded. The `'staff'` role is never touched by expiry logic.

## Role Assignment Rules

| Scenario                              | Role Set                                                 |
| ------------------------------------- | -------------------------------------------------------- |
| User registers                        | `'learner'` (default)                                    |
| User purchases Lite, webhook succeeds | `'lite'`                                                 |
| User purchases Pro, webhook succeeds  | `'pro'`                                                  |
| Subscription expires (lite/pro)       | `'learner'` (via auth middleware)                        |
| Admin manually assigns                | `'staff'` (via admin panel — unrelated to payment)       |
| User renews before expiry             | Role stays same, `subscriptionEndsAt` extends by 30 days |

## Fixes Applied

### Issue 1: One-time payment → 30-day subscription periods

**Problem**: Payment was a one-time transaction. No subscription period existed — the PricingModal showed "$X/mo" but never charged again.

**Solution**:

- Added `expires_at` to `orders` — set to `now() + 30 days` at order creation (`app/routes/payment.ts:44`)
- Added `subscription_ends_at` to `users` — set by the webhook on successful payment (`app/routes/PaymentWebhook.ts:58`)
- Renewals extend the existing `subscriptionEndsAt` by 30 days (pro-rata) rather than resetting
- Pricing copy changed from `"/month"` to `"/ 30 days"` (`PricingModal.tsx:173`)

### Issue 2: Permanent role upgrade → timed expiry with downgrade

**Problem**: Payment set `role = 'staff'` permanently with no way to downgrade.

**Solution**:

- Removed `'staff'` from the payment flow entirely — `'staff'` is now strictly a manually-assigned admin role
- Added two new subscription roles: `'lite'` and `'pro'` (`USER_ROLES` in `types.ts:24`)
- Webhook sets `role = 'lite'` or `'role = 'pro'` based on the purchased plan (`PaymentWebhook.ts:55`)
- Auth middleware (`middleware/auth.ts:26-28, 56-59`) checks `subscriptionEndsAt` on every request:
    - If expired and role is `'lite'`/`'pro'`, downgrades to `'learner'` in both context and DB
    - `'staff'` role is never touched by expiry logic
- `PaymentSuccess.tsx` loader refreshes the session cookie from DB so the new role is visible immediately without logout/login

### Issue 3: Misleading UI copy

**Problem**: PricingModal footer read _"Secure payment processing by Stripe"_ but the integration uses PayOS.

**Solution**: Changed to _"Secure payment processing by PayOS"_ (`PricingModal.tsx:219`).

## Resolved Issues

### Issue 1: Order code collision risk

**Problem**: `Date.now().slice(-9) + userId.slice(-1)` could collide for orders created in the same millisecond by users with matching last ID digits.

**Fix**: Replaced with `crypto.randomInt(100_000_000, 999_999_999)`. 900M possible values, negligible collision risk. (`app/routes/payment.ts:28`)

### Issue 2: Course access gating

**Problem**: Plan features (challenges/labs, AI chat) were not enforced — all authenticated users had equal access regardless of role.

**Fix**:

- Created `app/.server/payment/access.ts` — centralized helpers (`canAccessAI`, `getAIMessageLimit`, `canAccessChallenges`)
- **AI Chat gating** (`app/.server/chat/handler.ts`): checks role + daily quota (lite: 20/day, pro: unlimited). Tracks usage via `chat_messages` table with daily upsert. Returns 403 for learners, 429 when limit hit.
- **AI Chat button** (`app/layouts/MainLayout.tsx`): hidden for `learner` users
- **Challenge gating** (`app/pages/Challenges.tsx`): `canSubmit` flag in loader, action rejects learners with error. UI shows upgrade prompt with "View Plans" link (dispatches `open-pricing` custom event).
- **Pricing modal trigger**: `MainLayout.tsx` listens for `open-pricing` event, allowing the Challenges page to open the modal via cross-component event.
- **Chat API**: moved from Vite dev plugin to a React Router route (`app/routes/api.chat.ts`) for proper middleware/context support.

## Remaining Issues

1. **No cancellation flow** — Users cannot cancel their subscription early. Once paid, the 30-day access is granted regardless.
